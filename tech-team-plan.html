Part 1｜核心岗位需求明细

MVP 阶段核心原则：用最少的全职高手，配以适量的外包/兼职，换最快的验证速度。按「不可或缺」优先级排列，初期可由一人兼任多职，但职责必须有人覆盖。

5 Roles · MVP→A轮
1

后端开发工程师（架构师方向）

你的技术地基 · 不可或缺 ⭐
为什么不可或缺：产品不是简单 CRUD，包含高并发双向匹配算法、AI 模型调用与数据管道、支付与财务结算、严苛的隐私与数据安全要求。
  • 经验：5 年以上后端，主导过百万级用户产品的架构设计
  • 语言：Go / Java / Python（Go 更适合高并发场景）
  • 数据库：PostgreSQL（主数据）+ Redis（缓存与匹配队列）+ MongoDB（日志与行为数据）
  • 消息队列：Kafka / RabbitMQ（异步处理 AI 调用、通知推送）
  • 容器化：Docker + K8s（必须，从 Day 1 就上容器）
  • 推荐系统：有匹配引擎或推荐系统设计经验
  • 合规：熟悉数据隐私法规（GDPR/个保法），能设计数据隔离与脱敏方案
🚫 红线：没做过用户数据安全设计、不懂 API 版本管理的人不能用
2

前端/移动端开发工程师

用户的第一感知 · 不可或缺 ⭐
为什么不可或缺：核心交互是「左滑右滑」，对流畅度和动画要求极高——卡顿 = 流失。投资人/创业者群体偏高端，对 UI/UX 容忍度极低。
  • 经验：3 年以上移动端开发经验
  • 技术选型：Flutter 或 React Native（MVP 首选 Flutter，性能接近原生，一套代码覆盖两端）
  • 手势交互：必须有过复杂手势交互开发经验（拖拽、滑动、回弹动画）
  • 性能：理解移动端性能优化，60fps 是底线
  • IM 模块：有即时通讯模块集成经验（导师咨询需要文字/语音通信）
  • 加分项：做过社交类或卡片式交互产品的前端
3

AI/ML 工程师

你的核心护城河 · 不可或缺 ⭐
为什么不可或缺：这是区别于所有纯社交竞品的唯一壁垒。AI 财税法风控报告不是调用 ChatGPT 包装一下就行，需要结构化输出、风险点识别、文本理解。
  • 经验：2 年以上 NLP/文本处理方向经验
  • 语言：Python（必须）
  • 框架：LangChain / LlamaIndex 等 LLM 编排框架
  • Prompt：Prompt Engineering 深度经验——工程化、可评估、可迭代的提示流
  • 向量数据库：Pinecone / Milvus（项目文档语义检索）
  • 结构化输出：有过「非结构化文本 → 结构化报告」开发经验（合同审查、财报分析、尽调摘要等）
  • RAG 架构：理解检索增强生成，能降低模型幻觉
💡 MVP 务实做法：基于 GPT-4/Claude API 做精调 Prompt + 结构化输出，不要自研模型，把精力放在结果质量和准确率上
4

UI/UX 设计师

信任感的视觉化 · 不可或缺 ⭐
为什么不可或缺：独特挑战——「严肃的商业决策」与「游戏化交互」之间的张力。普通设计师会把 App 做成四不像。设计质感即信任度。
  • 经验：3 年以上移动端产品设计经验
  • 作品集：必须有过「复杂信息可视化」或「B 端工具类」设计案例
  • 信任设计：理解如何通过排版、配色、动效传递专业、安全、可靠
  • 加分：有社交或金融类产品设计经验
  • 交互能力：不只是画界面，能写交互说明文档
🎯 面试题建议：当场设计「一份 AI 风控报告的移动端呈现方案」，看信息层级和可读性
5

后端开发工程师（业务方向）

服务端功能实现者
为什么不可或缺：架构师定好基调和框架后，需要有人快速落地业务功能——用户系统、支付模块、IM、通知系统。
  • 经验：3 年以上后端开发经验
  • 技术栈：与架构师使用相同技术栈（便于协作）
  • 支付：有支付系统对接经验（Stripe/支付宝/微信支付）
  • 认证：有认证系统开发经验（身份认证、企业认证的多级审核流）
  • IM：熟悉 WebSocket / 长连接

Part 2｜技术联合创始人画像

技术联合创始人不是「最牛的码农」，而是能用技术翻译商业需求、用架构承载未来 3 年增长、用工程师文化吸引顶级人才的人。

Co-Founder Profile

🏗️ 硬性门槛

全栈视野 + 后端深厚

能从前端手势交互的 60fps，到后端匹配算法的复杂度，再到 AI 模型的调用链，全部心里有数。能精准判断每个环节的技术难度和人力投入。

做过 0 到 1

经历过一个产品从第一行代码到承载 10 万+ 用户的全过程，知道什么阶段该快速堆功能，什么阶段必须停下来重构。没经历过「技术债爆发」的人，会在错误的时间做错误的技术决策。

数据安全与合规的底层认知

必须能一开口就说出「BP 做存储端加密、传输层 TLS、数据库字段级脱敏、用户可见范围权限表……」——而不是等你提醒。

至少精通一种云原生生态

AWS / Azure / 阿里云其中之一，能估算成本。如果不能对「这个功能上线后每月会增加多少云成本」给出数量级估算，会烧光你的钱。

🧠 软性素质（这部分比技术更重要）

🎯 产品感，不是只接需求

你说「做一个 AI 风控报告」，他会反问：「报告是生成一页长图还是分步交互展示？用户最关心的 3 个风险点在哪？生成需要 15 秒，用什么动画填补等待焦虑？」——会挑战你的产品方案，而不是只会执行。

🔧 务实的技术选型能力

知道什么该自研、什么该买、什么该开源。匹配算法初期用「规则 + 简单向量相似度」就行，别上来就搞深度神经网络；AI 风控用 GPT-4 API 先跑通，别自建模型。

💰 对成本敏感

能从第一天就建立成本意识。能预判「某云函数方案流量上来后自动扩缩容三天烧光月预算」这类问题。

📣 工程师文化布道者

未来招 5 个、10 个、20 个工程师时，这个人的技术品位、代码审查习惯、文档规范程度，直接决定整个技术团队的水平。能把复杂技术问题讲清楚，才能帮你融资时过技术 DD。

🫡 抗压与极度务实

创业公司技术栈不可能完美，必须能在「屎山」上快速奔跑的同时，知道哪座屎山必须先铲掉。分清「完美主义」和「生存必须」。

🔍 如何找到这样的人？

  • 从哪里找：大厂 T7-T9 级别，做过类似社交/推荐/金融系统，且有副业或开源项目经历；另一个好来源是经历过一次创业退出（无论成败）的 CTO，他们对「什么不该做」有深刻理解。
  • 判断方法：先别让他写代码。给场景——「匹配池里 5000 个投资人标签 + 10 万个项目标签，首版匹配规则怎么设计才能让首屏推荐结果的双方感兴趣率达到 15%？」看分析框架是否从「冷启动」「多样性保证」「反馈闭环」这些商业结果出发。
  • 股权分配：技术联合创始人通常占联合创始人级别股份（15%-30%），取决于加入阶段和是否有薪资。天使轮后加入通常给薪资 + 可观期权。建议用 4 年兑现 + 1 年悬崖的标准方案保护项目。

Part 3｜团队搭建节奏建议

分三个阶段渐进式扩编，每个阶段目标清晰，避免过早烧钱。

Hiring Roadmap
正式启动前 原型验证

产品原型 + 高保真交互稿

CEO + 技术联创 + UI/UX 设计师（可兼职/外包），三股力量把产品概念锁定为可交互的原型。

👤 CEO 👤 技术联创 👤 UI/UX 设计师（兼职/外包）
天使轮 → MVP 3 人 · 3 个月

核心匹配 + AI 风控报告的最简版本必须上线

技术联创 + 1 名全栈（前后端都写）+ 1 名 AI 工程师，极致精简，只做验证必须的功能。

👤 技术联创 👤 全栈工程师 👤 AI 工程师
A 轮前后 完整团队

按清单补齐专职岗位

专职前端 + 专职后端业务 + 专职后端架构（如果技术联创本身能力覆盖架构，可先不招架构师，省下钱投市场）。

👤 技术联创 👤 全栈 → 拆分专职前后端 👤 AI 工程师 👤 专职前端 👤 专职后端（业务） 👤 架构师（视联创能力而定）
技术联合创始人不是「最牛的码农」，而是能用技术翻译商业需求、用架构承载未来 3 年增长、用工程师文化吸引顶级人才的人。 ——团队搭建核心原则
一句话总结：MVP 阶段 3 个人（联创 + 全栈 + AI），3 个月，必须上线核心匹配 + AI 风控报告的最简版本。之后再按 A 轮节奏补齐专职前端、专职后端业务、专职架构。
